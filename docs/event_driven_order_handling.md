# 事件驱动的报单/撤单处理方案评审与补全建议

## 目标

在“trade 请求只有报单(New)与撤单(Cancel)”的约束下，把策略执行改成**事件驱动**：以撮合回报/成交推送为主；当关键回报丢失时，用 REST 查询对齐真实状态，并通过“补发 update”推动状态机继续前进，最终保证：

- 策略状态可终结（成功完成或失败关闭），不会卡死在中间态
- 风险侧优先：对冲腿在异常下也能持续推进到“对冲完成/可接受的失败退出”
- 对外动作可控：避免重复下单/重复撤单导致的限频与状态紊乱

本文对你给出的流程做完备性检查，并给出需要补齐的关键点与可落地的规则化建议。

---

## 核心前提（你的方案隐含假设）

1. **Manager 每 100ms 扫描 strategy**：开仓态超时会触发撤单事件；撤单失败时会持续触发撤单。
2. **Engine 最终会回报**：除了链路断裂/长时间丢包，撤单/报单的响应最终能回来。
3. **允许通过 REST 查询“订单真实状态”**：用来修复 WS 推送缺失（trade/cancel 回报丢失等）。

只要以上假设被破坏（例如：REST 也持续失败、WS 断链、频率限制严重），就需要额外的“降级/熔断/人工介入”出口，否则策略可能无限重试并放大风险。

---

## 事件与状态机（建议明确化）

建议显式区分**事件来源**与**状态推进**，并保证幂等：

- 事件来源
  - `MgrTick`：100ms 扫描产生的“超时撤单/继续对冲”等事件
  - `EngResp`：报单/撤单 REST 或 WS 的请求响应（ack / reject）
  - `OrderUpdate`：订单状态更新（new/partially_filled/filled/canceled/rejected/expired…）
  - `TradeUpdate`：成交明细（可多笔、可部分成交）
  - `RestQueryResp`：REST 查询结果（用于补齐/对账）
- 状态机关键原则
  - **单向推进**：状态只能向“更确定/更终结”方向走（例如：`WORKING -> FILLED`，不能从 `FILLED -> WORKING`）
  - **动作幂等**：同一个 `CancelIntent` 不应并发触发多次撤单；同一个 `HedgeIntent` 不应重复下单导致过度对冲
  - **补发 update 也要去重**：REST 补齐的 `TradeUpdate/OrderUpdate` 可能与后到的 WS 推送重复

## 开仓侧（Open leg）

### A. 报单（New）

你写的“开仓侧报单：直接关闭 strategy”，建议解释为：

- **关闭信号生成/策略逻辑**，进入“等待开仓回报/成交”的被动态（避免重复发起开仓）
- 仍然保留状态机与风控动作（例如超时撤单、对冲触发等）

如果你本意是“开仓报单失败就直接关闭 strategy”，请在实现里区分：

- `NewRejected`（参数错误、余额不足等确定性失败）→ 直接关闭/标记失败
- `NewUnknown`（网络/超时未收到 ack）→ 进入“待确认”并触发 REST 查询对账

### B. 撤单失败：原因分类与处理

你列出的可能原因（1~4）是合理的，但需要补齐“部分成交”等中间态，并澄清第 3 类的动作：

#### 1) 已完全成交，但漏掉成交推送（订单已不存在，无法撤单）

- 动作：`REST Query(order_id)` → 若返回 `filled` 或成交明细：补发 `TradeUpdate`（含累计成交量/均价/手续费等），然后触发对冲。

#### 2) 已 cancel，但丢失 cancel 回报（订单可能已不存在或状态已是 canceled）

- 动作：`REST Query(order_id)` → 若返回 `canceled`：补发 `OrderUpdate(canceled)`，然后触发对冲（如果你的流程是“开仓撤单后仍需要对冲剩余成交量”，则对冲量应为**已成交量**而不是原始下单量）。
- 你原文里出现“2 => 直接关闭 strategy”，这与“补发 cancel 并触发对冲”存在冲突；建议以**REST 查到 canceled 后推进状态机**为准，而不是直接关闭。

#### 3) 报单就失败了 / 本地无 order_id

- 若“无 order_id（根本没挂上）”：撤单无需处理，应直接把该 `CancelIntent` 视为“已满足”（因为没有可撤的对象），并让策略进入可终结状态（失败关闭或回到可重试的上游状态，取决于你的业务）。
- **不建议“重新撤单”**：没有 order_id 的重复撤单只会徒增噪音与限频风险。

#### 4) REST/请求侧错误（网络、限频、超时）

- 动作：进入“待确认/待重试”分支，但需要**限频与退避**：
  - 重试间隔指数退避（例如 200ms/500ms/1s/2s…）并加随机抖动
  - 每个 venue 单独的速率预算（避免多个策略叠加把 REST 打爆）
  - 超过上限后进入 `StrategyStuck`/`ManualIntervention`（至少停止继续下单，只保留风控/告警）

### C. 需要补齐的关键中间态：部分成交

撤单失败/撤单结果未知时，最常见的真实状态之一是 `partially_filled`。注意这里我用的 `working` 是**概念态**：表示订单仍为 open/live（未 `canceled/filled/expired/rejected`），还在簿上、还能继续成交、也仍然“可撤”。

建议完全基于 `cum_filled_qty`（累计成交量）驱动补齐与对冲：

- REST 查到 `status=open/live` 且 `cum_filled_qty > local_cum_filled_qty`：
  - 先补发 `TradeUpdate(delta = cum_filled_qty - local_cum_filled_qty)`（或等价地把本地 `local_cum_filled_qty` 更新到 REST 返回值）。
  - 对冲量永远按 `need_hedge_qty = cum_filled_qty - cum_hedged_qty` 计算（`cum_hedged_qty` 为对冲侧累计已对冲量）。
- 若此时你仍处于“希望撤单”的分支（例如开仓超时触发撤单）：
  - 在**不并发**、带退避的前提下，再发送一次 `cancel(order_id)` 是合理的（目的是阻止继续成交），直到 REST/推送确认订单进入终态。
- REST 查到 `status=canceled` 且 `cum_filled_qty > 0`：
  - 补发 `TradeUpdate` + `OrderUpdate(canceled)`，然后仅对冲 `need_hedge_qty = cum_filled_qty - cum_hedged_qty`，完成后结束。
- REST 查到 `status=filled`：
  - 补发 `TradeUpdate`，然后按 `need_hedge_qty` 完成对冲并收尾。

## 对冲侧（Hedge leg）

### A. 对冲侧开仓失败（下单失败）

你给出的规则方向正确，但建议补齐“风控导向”的降级：

- `post_only` 挂单失败：重新挂单（通常需要重新定价到更保守的 maker 价位）
- 其他错误：继续挂单但打印 error log（限频等）
  - 建议明确：**不能在高频循环里无限重试**，需要 cooldown/退避与“最大尝试次数/最大持续时间”。
  - 建议加降级策略：若多次 maker 失败且敞口在扩大，可切到 taker（IOC/market）以尽快把风险打掉（由配置控制）。

### B. 对冲侧撤单失败：原因分类与处理

你给出的处理框架可用，建议同样补齐部分成交与“未知状态”的出口：

#### 1) 完全成交但漏掉成交推送

- `REST Query(order_id)` → 查到 `filled`/成交明细：补发 `TradeUpdate`，更新已对冲量，关闭 strategy。

#### 2) 已 cancel 但丢失回报

- `REST Query(order_id)` → 查到 `canceled`：补发 `OrderUpdate(canceled)`，按“剩余未对冲量”继续对冲。

#### 3) 订单挂单就失败 / 本地订单不存在

- 与开仓侧一致：无 order_id 时撤单应直接视为完成。

#### 4) REST/请求侧错误（网络、限频、超时）

- 你写的“3、4 => query 盘口，重新触发对冲”可以作为业务动作，但仍需：
  - 先保证**不会重复对冲已成交量**（靠累计已对冲量/净敞口校验）
  - 对“撤单未知”进行节流：避免撤单失败后立刻再次下单导致同方向叠加

### C. 关于 timeout（你结论基本成立，但需加保护）

你认为“mgr 100ms 扫描 + 持续触发 cancel，因此不需要额外 timeout”，在工程上可成立，但需要两类保护：

- **in-flight 防抖**：同一订单的 cancel 请求未完成前，不要并发发起第二次 cancel（否则回报乱序、限频更容易触发）。
- **无限重试的熔断**：若连续 N 秒 cancel/query 都失败，策略应进入“降级安全态”（停止新下单，只保留最小风控动作 + 告警），否则会在链路断裂时把 REST 打到雪崩。

## 完备性检查清单（建议补齐/明确的点）

1. **幂等与去重规则**
   - WS 推送晚到、REST 补发早到时如何去重（建议以 `order_id + last_update_id/seq` 或 `(order_id, status, cum_filled_qty, ts)` 组合去重）。
2. **部分成交的统一语义**
   - 对冲量以 `need_hedge_qty = cum_filled_qty - cum_hedged_qty` 为准，而不是以原始订单量为准；`cum_filled_qty` 以 REST/推送对齐并单向递增。
3. **竞态处理**
   - “撤单请求成功，但随后收到成交推送”的乱序是常态；状态机必须允许 `canceled` 之后仍收到 `trade` 并以成交为准更新累计成交。
4. **REST 查询节流与退避**
   - 按 venue/account 维度做队列与速率预算；失败做指数退避与抖动。
5. **链路断裂/交易所故障的出口**
   - WS 断链 + REST 限频时，策略必须能进入可观测的 `Stuck/Degraded` 状态并停止扩大风险。
6. **风控降级策略**
   - 对冲腿多次失败时是否允许 taker、最大允许滑点、最大对冲持续时间等（建议配置化）。
7. **可观测性**
   - 统计“REST 补发次数、WS 缺失率、撤单失败率、重试次数、对冲完成耗时”，并对异常阈值告警。

## 结论

你的方案在“WS 偶发丢包、撤单失败需对账”的主线上是成立的，核心思路（**撤单失败→REST 查真实状态→补发 update 推动状态机**）也是正确的。

主要不完备点集中在：

- **部分成交**与**乱序/重复事件**未明确（容易导致对冲量计算错误或重复对冲）
- **无 order_id 的撤单**动作需要统一为“直接视为完成”（避免无意义重试）
- **REST/限频/断链场景的节流与熔断**需要补齐（否则在极端情况下会无限重试并放大风险）

把上述清单补齐后，这个事件驱动方案就能更稳定地覆盖真实交易所环境中的常见异常。
